# Сводка для разработки

Техническая выжимка из [BOT_FLOW_MVP](../03.%20Scenarios/BOT_FLOW_MVP.md) — сущности, действия бота, LLM, алгоритмы.

## Контексты организатор vs участник

Пользователь может быть организатором одной встречи и участником другой. Состояния разделены по `meeting_id`: `user_states` — только для флоу организатора; `participant_selection` и `participants` — для участника.

---

## Сущности

| Сущность | Поля |
|----------|------|
| **Встреча** | id, название, слоты (массив: дата+время, от LLM), статус (создана / время назначено), чат_id (где создана), создатель_id |
| **Слот** | дата, время (всегда в будущем); у участника: отмечен как удобный или не отмечен (считается занят) |
| **Участник** | user_id, встреча_id, статус (отметил слоты / отказался «не смогу» / не ответил), ответы — только удобные слоты (слот_id в списке выбранных). *В MVP в participants попадают только ответившие (replied) и отказавшиеся (declined); счётчик «ещё не ответили» в сводке недоступен.* |

## Ключевые действия бота

1. **Команда /start** (или «Start» в меню) — сбросить контекст FSM и показать приветствие с кнопкой «Давай соберёмся!». Исключение: `/start meeting_xxx` — deep link для участника.
2. Принять название и опции для встречи (слоты) от организатора — **текстом**; LLM парсит и возвращает слоты в нужном формате.
3. При неудачном парсинге — уточнить у пользователя.
4. Создать встречу, сгенерировать ссылку `t.me/bot?start=meeting_{id}`.
5. В групповом чате — запостить сообщение с кнопкой.
6. По ссылке — сразу показать слоты и режим ответа: участник отмечает только удобные слоты (неудобные не трогает) или нажимает «не смогу».
7. Сохранить ответы (слоты или отказ).
8. Упорядочить слоты по количеству отметивших (сначала слоты, где все отметили как удобный).
9. После выбора организатором — спросить место. Свободным отправить полное уведомление; занятым в выбранном слоте — «Сможешь прийти?»; после ответа — подтвердить или вывести из списка.

## Интеграция LLM

**Ключевая фича MVP:** организатор вводит опции для встречи **естественным языком** — LLM (OpenAI gpt-4o-mini) парсит и возвращает слоты. В коде бота **никакой** обработки дат — только LLM.

**Назначение:** Распознавать текст организатора и возвращать готовый список слотов в формате, пригодном для бота.

**Что парсит LLM:**
- Один шаг: «какие опции предложить участникам»
- Примеры ввода: «суббота 12:00, 15:00, 18:00»; «в эту субботу»; «25 февраля вечером и 26-го днём»; «следующие выходные с 12 до 18 каждые 2 часа»

**Ограничение:** Все слоты должны быть в будущем. Нельзя назначить встречу в прошлом. LLM отклоняет даты в прошлом; бот дополнительно фильтрует слоты с `datetime` в прошлом.

**Формат ответа LLM:** JSON: `{ ok: true, slots: [{ date, time, datetime }, ...] }` или `{ ok: false, error: "..." }`. LLM возвращает слоты в формате, готовом для использования ботом. При невозможности распознать — `ok: false`, бот запрашивает уточнение.

---

## Алгоритм показа слотов организатору

Слоты упорядочиваются по удобству для группы:
1. **Слоты, где все ответившие отметили как удобный** — в начале списка.
2. **Остальные слоты** — по убыванию числа отметивших.

Для каждого слота показывать: сколько ✓ отметили как удобный, сколько ✗ не отметили. Организатор может выбрать любой слот. Участникам, которые не отметили выбранный слот, уходит вопрос: «[Название] назначена на [дата, время]. Сможешь прийти?»
